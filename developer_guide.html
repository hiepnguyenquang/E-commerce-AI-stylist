BFF & Microservice Hybrid

Next.js tương tác trực tiếp với MedusaJS đóng vai trò BFF (Backend-for-Frontend). Medusa xử lý nghiệp vụ thương mại nặng và điều phối công việc cho FastAPI AI thông qua RabbitMQ.

Asynchronous Queue & SSE

Tiến trình ghép đồ AI (CatVTON) tốn 3-5 giây trên GPU được đưa vào RabbitMQ bất đồng bộ. Client duy trì một kết nối Server-Sent Events (SSE) để nhận cập nhật trạng thái vẽ thời gian thực.

Hybrid Vector Search

Tìm kiếm ngữ nghĩa (RAG) sử dụng kho sản phẩm nhúng CLIP vector được lưu trữ ngay trong LanceDB (Serverless Vector Database), mang lại phản hồi đề xuất trang phục vượt trội.

1 Thiết lập thông số & Tải trang phục

Người dùng thiết lập chiều cao, cân nặng và tải lên ảnh cá thể để làm nền thử đồ. Dữ liệu được quản lý đồng bộ nhờ cơ chế Hydration API mới:

  • Giao diện Canvas thực hiện crop tỷ lệ 3:4, nén ảnh xuống dưới 2MB để tối ưu băng thông.
  • API `GET /api/v1/ai-profile` tự động nạp lại thông số người dùng từ Database mỗi khi tải lại trang, chấm dứt phụ thuộc LocalStorage.
  • Tủ đồ cá nhân hỗ trợ tải lên trang phục, tự động gọi thư viện tách nền xử lý đưa về dạng trong suốt (`PNG`).
  • Đồng bộ Xóa hoàn chỉnh: Khi click xóa trang phục, hệ thống xóa DB record, xóa file vật lý trong `.medusa/uploads` và gửi RabbitMQ để dọn dẹp Vector nhúng của trang phục đó trong LanceDB.

2 RAG AI Stylist & Thay thế Đồ (Replace Item)

Người dùng gửi câu hỏi thời trang dạng ngôn ngữ tự nhiên. AI phân tích, tìm kiếm vector sản phẩm trong LanceDB và kết hợp Gemini LLM gợi ý Set đồ (Outfit).

  • Bộ lọc Ý định (Intent Guardrail): AI lọc trước câu hỏi của user. Nếu không liên quan thời trang (VD: thời tiết), hệ thống từ chối khéo bằng JSON `{"refusal": true, "message": "..."}` và bỏ qua tìm kiếm Vector để tiết kiệm tài nguyên.
  • Hydration Sản phẩm: Dữ liệu thô gửi từ AI Python (chỉ gồm mảng ID) sẽ được backend Medusa chặn lại, tiến hành Hydrate chi tiết (Tên, Ảnh, Giá) từ Database PostgreSQL trước khi trả về Frontend, đảm bảo hiển thị tức thì không độ trễ.
  • Đổi Món Đồ (Replace Item): Nút "Đổi món khác" trong thẻ Set đồ cho phép tìm các sản phẩm tương đồng về Style/Category bằng cách gọi API `/v1/stylist/replace` thực thi tìm kiếm tương đồng vector, thay thế phần tử đó trong Set đồ mà không phá vỡ tổng cục.

3 Ghép đồ bất đồng bộ qua Queue & SSE (Try-On)

Người dùng có thể chọn thử Set đồ (gồm 1 Áo + 1 Quần/Váy) lên ảnh nền cơ thể. Quá trình xử lý phức tạp của Model CatVTON được điều phối mượt mà:

  • Thử đồ đa lớp (Multi-step VTON): Hệ thống tạo luồng Job 1 (ghép Áo) đẩy vào RabbitMQ. Khi AI Service hoàn thành Job 1, nó lấy URL kết quả làm ảnh gốc (`human_image_url`) để tạo tiếp Job 2 (ghép Quần) có liên kết `parent_job_id` với Job 1.
  • Không phụ thuộc SSE connection: Tiến trình lắng nghe hàng đợi RabbitMQ (`ai_vision_results`) được tách khỏi API SSE và chạy trực tiếp ở file Loader Node.js ngay khi bật server. Dù client mất mạng/mất kết nối SSE, job AI vẫn tự động hoàn tất và lưu DB.
  • Server-Sent Events (SSE): Cung cấp cập nhật trạng thái thời gian thực đến Client (`pending` → `rendering` → `completed`).
  • VTON Result Gallery: Người dùng có thể nhấn nút "Lưu kết quả" để lưu ảnh thử đồ vào danh mục Outfit riêng trong Tủ đồ.
AI Python Service Worker Consumer: 
# apps/ai-service/src/worker.py (CatVTON Consumer)
async def process_vton_job(ch, method, properties, body):
    payload = json.loads(body)
    job_id = payload["job_id"]
    human_img = payload["human_image_url"]
    garment_img = payload["garment_image_url"]
    category = payload["category"] # upper_body / lower_body / dress
    
    # 1. Chạy CatVTON Pipeline (Persistent in VRAM)
    result_path = pipeline.predict(human_img, garment_img, category)
    result_url = upload_to_medusa_storage(result_path)
    
    # 2. Publish kết quả lại RabbitMQ
    publish_result_to_queue({
        "job_id": job_id,
        "status": "completed",
        "result_image_url": result_url
    })

4 Đặt hàng nâng cao (Saga Checkout Workflow)

Đảm bảo tính nhất quán dữ liệu thương mại bằng cách sử dụng cơ chế MedusaJS v2 Workflows (Saga Pattern) để quản lý luồng thanh toán và tồn kho:

  • Bước 1 (reserveInventory): Khóa (tạm giữ) số lượng tồn kho sản phẩm trong giỏ hàng để tránh bán vượt mức.
  • Bước 2 (createOrder): Khởi tạo đơn hàng chính thức kèm metadata theo dõi xuất xứ từ AI (được gợi ý bởi Stylist hay thử đồ).
  • Bước 3 (processPayment): Gọi cổng thanh toán hoặc giả lập thanh toán an toàn. Nếu thất bại, kích hoạt cơ chế bù trừ (Compensating action) để mở khóa tồn kho.
  • Đồng bộ Chuyển đổi (Conversion Tracking): Đơn hàng thành công sẽ tự động cập nhật lại lịch sử `stylist_sessions` để ghi nhận và tối ưu hóa các đề xuất AI tương lai.
Cơ cấu Workflow Saga trên Medusa v2: 
// apps/api-medusa/src/workflows/checkout-saga.ts
import { createWorkflow, StepResponse } from "@medusajs/framework/workflows-sdk";

export const checkoutSagaWorkflow = createWorkflow(
  "checkout-saga",
  function (input: CheckoutInput) {
    // 1. Giữ tồn kho
    const inventory = reserveInventoryStep(input.items);
    
    // 2. Tạo hóa đơn đơn hàng
    const order = createOrderStep(input);
    
    // 3. Tiến hành thanh toán
    const payment = processPaymentStep({ order, paymentInfo: input.payment });
    
    return { order, payment };
  }
);

Chi tiết Model AI & GPU Optimization

Cấu hình tối ưu để chạy các tác vụ sinh ảnh VTON nặng trên GPU thế hệ mới một cách mượt mà.

PERSISTENT LOADING

Persistent VRAM Configuration

Do các tác vụ NLP đã được chuyển dịch hoàn toàn lên Cloud API (Gemini 1.5 Flash 8b), toàn bộ VRAM của RTX 5060 được dành riêng cho mô hình sinh ảnh VTON.

  • Mô hình CatVTON Adapter sử dụng định dạng bfloat16 siêu nhẹ.
  • Model được nạp thẳng vào VRAM một lần duy nhất lúc khởi động Server Python (Persistent loading), loại bỏ hoàn toàn độ trễ 4-6 giây nạp/giải phóng model trên mỗi request.
WINDOWS & WSL2 IPC

WSL2 Networking Gateway

Môi trường chạy AI Service là Python chạy trên WSL2 (Ubuntu), trong khi Medusa Core chạy trực tiếp trên Windows Host. Hai môi trường giao tiếp chéo qua địa chỉ Gateway của card mạng ảo WSL2 (lấy tự động bằng lệnh ip route show default) thay vì địa chỉ 127.0.0.1 cục bộ của Windows.

Tuning PyTorch/CUDA cho RTX 5060 (SM_120):

Các dòng card đồ họa Blackwell mới (RTX 5060 / SM120) chưa có file biên dịch sẵn cho PyTorch 2.6 ổn định. Hệ thống sử dụng cơ chế kích hoạt JIT biên dịch thời gian thực trong môi trường .env: 

# Cấu hình biến môi trường ép chạy PTX JIT
TORCH_CUDA_ARCH_LIST="12.0"
CUDA_MODULE_LOADING=LAZY

# Sử dụng UV cài đặt PyTorch Nightly hỗ trợ CUDA 12.8
uv pip install --pre torch torchvision \
  --index-url https://download.pytorch.org/whl/nightly/cu128
⚠️ Lưu ý symlink trên Windows: Khi tải weights của các snapshot model HuggingFace về Windows Host, hãy chắc chắn hệ điều hành của anh đã bật **Windows Developer Mode** để cho phép Python tạo các file symlink liên kết, tránh lỗi phân quyền tải weights.

Xử lý Sự cố & Hỏi đáp nhanh (FAQ)

Danh sách các lỗi thường gặp trong quá trình tích hợp và giải pháp khắc phục triệt để.

Lỗi: Backend Medusa bị crash hoặc restart liên tục khi có người dùng upload ảnh?

Nguyên nhân: Ban đầu, ảnh upload được lưu trong thư mục public/uploads của Next.js hoặc Medusa. Trình biên dịch Nodemon/File Watcher phát hiện có file mới được tạo và tự động kích hoạt tiến trình hot-reload khởi động lại server Node.
Khắc phục: Đã chuyển thư mục lưu trữ sang .medusa/uploads/ nằm ngoài vùng quét của File Watcher. Ảnh vẫn được phân phối bình thường qua static routing middleware của Medusa tại URL `/uploads/*`.

Lỗi: Thư viện `@imgly/background-removal-node` báo lỗi "Unsupported protocol: d:" trên Windows?

Nguyên nhân: Thư viện tách nền WASM của Imgly không thể parse các đường dẫn tuyệt đối dạng ổ đĩa vật lý của Windows (VD: D:\project\...).
Khắc phục: Sử dụng hàm pathToFileURL từ thư viện url của Node.js để ép kiểu đường dẫn về định dạng URI chuẩn quốc tế trước khi đưa vào hàm xử lý tách nền: file:///D:/project/....

Lỗi: AI Stylist Chat trả về các set đồ gợi ý bị vỡ ảnh hoặc không thể nhấn thêm vào giỏ hàng?

Nguyên nhân: Do AI Stylist trả về gợi ý phối đồ kết hợp cả sản phẩm của cửa hàng (Store Products) và trang phục cá nhân của tủ đồ người dùng (Closet Items). Đồ cá nhân không thể thêm vào giỏ hàng và có URL ảnh nội bộ của user.
Khắc phục: Thẻ gợi ý OutfitCard đã được cập nhật logic kiểm tra phân loại `type`. Nếu là đồ cá nhân (`type === 'closet'`), giao diện sẽ hiển thị nhãn **"Tủ Đồ"**, tự động ghép đúng host URL backend để hiển thị ảnh, và **ẩn nút "Thêm vào giỏ"**.
